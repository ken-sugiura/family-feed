## グリルで確認した理解

> AI との Q&A で確認した「自分の言葉で説明できる状態」の記録

| 問い | 自分の答え | 判定 |
|------|-----------|------|
| 今の `data/*.json` 方式のどこが不便か？ | リロードで消える・妻の投稿が届かない・写真が保存できない | ✅ |
| リポジトリとDBの違いは？ | リポジトリ＝コード、DB＝ユーザーが作るデータ | ✅ |
| 写真ファイルはどこに置く？ | ストレージサービス（S3・Amazon Photos 等）の専用置き場 | ✅ |
| DBには何を入れる？ | 写真の URL ＋ 投稿日時・キャプション・子どもタグなどのメタデータ | ✅ |
| リポジトリには何を入れる？ | コード＋設定ファイル＋デモ用サンプルデータ（個人情報は入れない） | ✅ |

## 現状（今のファミリーフィード）

イベント・子ども・カテゴリのデータが `data/*.json`（リポジトリ内）に置かれている。

**問題点**
- ページをリロードすると追加・編集した内容が消える
- 妻が投稿しても自分の画面に届かない（リアルタイム共有できない）
- 写真ファイルをアプリ内に保存できない

## 判断の基準（自分の言葉で）

| 問い | 答え |
|------|------|
| これはコードか・データか？ | コード → リポジトリ／データ → DB |
| ユーザーが変更するものか？ | Yes → DB、No → リポジトリ |
| ファイルサイズが大きいか？ | Yes（写真・動画）→ ストレージ |
| リアルタイムで複数人が使うか？ | Yes → DB |
| 個人情報・秘密情報を含むか？ | Yes → リポジトリに入れない（.env 等） |

---

## 次のステップ（実装するなら）

1. **Supabase** を使って `events` / `children` / `categories` テーブルを作る
2. `data/*.json` の読み込みを Supabase クライアントの fetch に置き換える
3. 写真は Amazon Photos の共有 URL を DB に保存する（現状維持で OK）
4. ユーザー認証を追加して家族だけがアクセスできるようにする
